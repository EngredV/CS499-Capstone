Narrative

As you're already aware, I decided to use only one artifact across this whole capstone project, and I decided to build OrnaFlow from the ground up. I chose to build this inventory web app for this section of the capstone because it showcases my ability to build a complex, real-world app that involves integrating both frontend and backend technologies completely from scratch. The creation of this app will help demonstrate my ability in developing full lifecycle apps which ranges from designing not only the UI of the web app but also the database schema, API routes in FastAPI, implementing efficient data fetching and even UI filtering as well. OrnaFlow reflects not just my technical ability but also my practical problem-solving and project management skills.

Therefore, continuing to build upon OrnaFlow, I focused on implementing most of the algorithms and data structures requirement for this section which involved enhancing OrnaFlow with multi-criteria filtering, real-time search, sorting mechanisms, and alert generation logic. Yes, I met the course outcomes that I planned to meet with this enhancement section. I had proposed to enhance four things for this section – search and filtering, low inventory alerts, sorting algorithm for quantity and expiring ingredients and products, and batch expiration logic as well. Out of these four, I achieved the first three and the fourth one I will integrate after submitting the project which is the batch expiration logic. I wasn’t able to do so due to time, and I chose to prioritize other functions and features that were more urgent to tackle.

For the first enhancement – search and filtering, I set it up so that I would be able to search inventory and alerts by partial name (first two-three letters), category, ingredient/item, status, and even message. I used regex pattern filtering along with an array filter in order to be able to implement this first feature. I even managed to add a feature where the words (from result) that match what was searched are highlighted. I will be following my professor's feedback to be more specific in my narrative, and I will be implementing more details into my narrative as a result.

  • Data Structure used: Arrays of objects from fetched data and it's processed in-memory.
  • Algorithmic Strategy: Filtering with .filter() and regex pattern filtering.
  • Time Complexity:
    • Regex filtering: O(n) where n is the number of items that are on the list.
    • Since the operation is fast for small to moderate datasets, which is the case for my use case, it is ideal for the expected scale of OrnaFlow.
  • Optimization Considerations: I plan to implement pagination and even backend-based filtering when the dataset grows in the future.

The second enhancement involved implementing low inventory alerts by using a threshold-based algorithm, the goal here was to flag items whose quantity fall below a minimum threshold that we need to input. The example I used to test if my implementation was successful was setting Lavender Essential Oil’s minimum to 128 oz and I put that Lavender Essential Oil has 127 oz in stock. As expected, the system picked up on the low threshold and triggered a yellow triangular warning icon next to the stock available and also changed the status tag it had to a low stock one. On the alerts page an alert was created notifying me that Lavender Essential Oil is below its threshold.

  • Data Structure used: plain JavaScript objects in order to represent inventory items. The alerts are stored in an array and rendered in a table.
  • Time Complexity: O(n) for checking each item (so per item check) and again, where n is the number of items that are on the list.

For the third enhancement I made in this second part of the capstone was the sorting algorithm for quantity and expiration by using custom sorting algorithms. This lets me sort by the quantity of the ingredient and soap bars. As well as their expiration dates. I briefly tested both the sorting by expiration dates from soonest to expire first and latest to expire first and sorting by quantity from lowest to highest and from highest to lowest with the small amount of data I had.

  • Data Structure used: again, used inventory items as objects in an array and sorted in place.
  • Time Complexity: O(n log n) since this is what JavaScript's built-in .sort() time complexity typically is for average and worst cases.

While implementing this section of the capstone, I found that I deepened my understanding of state-driven UI development. More specifically, when it comes to complex user interactions such as multi-filtering, search, and selection. I gained practical experience with implementing and testing performance efficient data filtering, sorting, and conditional logic implementation not only for my real-world business requirements but that can also be applied to other real-world business requirements as well.

A key challenge that I faced in this section of the capstone was definitely with the filtering logic. Getting the filtering system to support multi-select and sync with all of the UI features I have going on such as the pills, and filter selection modal, required very careful state management and much frequent debugging. I also had to ensure that data remained consistent across multiple components while also maintaining performance and a clean UI/UX.

Narrative

Again, I decided to use only one artifact across this whole capstone project, OrnaFlow. I chose to build the OrnaFlow web app for this section of the capstone because again, it allows me to demonstrate my ability to build a complex, real-world app involving a fullstack integration. Remember, that the original artifact that I chose was an event-tracking mobile application that I developed for my CS360: Mobile Architecture and Programming course built in JAVA using Android Studio. This app allows users to create, edit, view, and delete events that have titles, dates, times, and descriptions, that is, normal CRUD operations. All of its data is stored locally via SQLite for simple local data storage. However, I quickly realized that it lacks scalability and the ability to handle multi-user interactions which is important. Which is why I also selected this item because I wanted to showcase my skills in upgrading and integrating PostgreSQL as the chosen database for my new inventory tracking web app - Ornaflow. This implementation and setting up from scratch demonstrate how much both my understanding and application of database systems in a real-world project or setting has advanced. Other abilities that this improvement showcases is my ability to design and normalize complex relational schemas, implement foreign key constraints, and also structure query relationships that are efficient. In this case, I did it for inventory items, batches, status tags, categories, units, and even batch ingredients tables. By integrating PostgreSQL with FastAPI my fullstack proficiency in not only managing asynchronous database queries, handling ORM relationships, and also ensuring that the data integrity is kept across users.

To break it down more, I designed a normalized relational schema in order to be able to fully represent complex business logic such as batch tracking, ingredient usage, and stock statuses. I also implemented JOIN queries so that I could efficiently fetch related data such as units, ingredients, and even the soap bars themselves, and finally, handling CRUD operations via SQLAlchemy ORM. My professor's feedback was positive for this section and so I focused on ironing out some final details of the app overall, finish getting everything connected overall, adding more test data (seeding data), and prepping for further features that I will add after submitting the project. Yes, I met the course outcomes that I planned to meet with this enhancement. I successfully migrated and or implemented PostgreSQL and Fast API. As for what I feel I have left to implement, I still need to finish creating a modal to input individual soap bar information, images, and tracking. I also need to integrate both the barcodes and qr codes in a way where I can scan them with my phone. As well as adding logic for the system to automatically calculate ingredient usage and available stock instead of having to update it manually myself. Same for the soap bars too.

As I was creating the database and improving it, I learned how to design relational database schemas in order to fully accommodate my evolving business needs while also making sure that redundancy is avoided and most importantly, keeping consistency. Upon reflecting, I realized that the most critical lesson I had through this section of the capstone is how important it is to truly understand schema relationships since we can have one to many or many to many. This is important because we need to keep in mind and even estimate how this will directly impact query complexity, as well as data retrieval patterns. One of the challenges that I came across was mismatches between the ORM models and the API schemas. This was especially true when having to deal with nested relationships. For this I had to implement foreign keys, manage join tables, and properly serialize the data so that no raw ORM instances are leaked.
